# swagger-ts-mcp v0.1.3版本检查与下版本功能优先级

> 检查日期：2026-03-31
> 当前 npm latest：`0.1.3`

##1) 当前版本健康度检查结论

### 已验证项

- 构建通过：`npm run build`
- 测试通过：`npm test`（7 个测试文件，109 条用例全部通过）
- 打包检查通过：`npm pack --dry-run`
- npm 发布状态正常：`dist-tags.latest =0.1.3`

###发现的问题（按影响排序）

#### P1：发布包包含测试产物，包体可进一步收敛

-现象：`npm pack --dry-run` 显示 `dist/src/__tests__/*.js` 与 `*.d.ts` 被一起发布。-影响：

- 增加包体积和安装成本；
- 用户侧不需要测试编译产物。
- 建议：
- 在 `tsconfig` 或构建流程中排除 `src/__tests__`；
- 或在发布前增加清理步骤，确保 `dist` 不含测试产物。

#### P2：CLI 输出包含 emoji，自动化环境可读性一般

-现象：`bin/index.ts` 中有 `📄 / ✅ / ⚠️` 等输出。-影响：CI 日志/纯文本终端可读性有时受影响。

- 建议：提供 `--no-emoji` 或 `--log-format plain|pretty`。

#### P2：写入逻辑存在可清理的小冗余

-现象：`src/writer.ts` 中 `const output = endsWithNewline ? lines.join('\n') : lines.join('\n');` 两分支完全一致。-影响：功能无影响，但可读性一般。

- 建议：简化表达，避免误导后续维护者。

---

##2) 下一个版本功能优先级（建议）

## P0（下一版本必须做）

###1. 发布产物瘦身（不发布测试编译结果）

- 用户价值：直接降低安装体积，提升专业度。
- 实现成本：低。
- 风险：低。
- 建议版本：`0.1.4`（patch）。

###2. 明确失败退出码与错误分层输出 -目标：当存在关键错误（例如 Swagger 拉取失败、文件解析失败）时，CLI退出码和日志结构可预测。

- 用户价值：便于 CI/CD 集成。
- 实现成本：低-中。
- 风险：低。

---

## P1（下一版本强烈建议做）

###3. 增加 `--functions`（CLI）与 README 对齐

- 背景：MCP 已支持 `functionNames`，CLI侧可以补齐等价能力。
- 用户价值：增量改造时可只处理目标函数，避免全文件改动。
- 实现成本：中。

###4. 增加幂等/重复插入保护测试（端到端）-目标：多次执行同一命令不应重复插入类型定义。

- 用户价值：提升稳定性信心。
- 实现成本：中。

###5. 更细粒度的输出模式

- 建议参数：`--silent`、`--json`。
- 用户价值：便于脚本消费与 IDE 集成。
- 实现成本：中。

###6. 支持本地 Swagger/OpenAPI JSON 文件输入（离线/内网场景）

-现状：当前 `--swagger`仅按 URL 拉取，遇到需登录认证或内网权限限制时可能失败。

- 建议能力：支持 `--swagger`传本地 JSON 文件路径（如 `./openapi.json`），或新增 `--swagger-file` 参数。
- 用户价值：在无法直接访问在线文档时，仍可通过导出的 API JSON 正常生成类型。
- 实现成本：中。-备注：该能力与现有在线 URL方式并存，保持向后兼容。

---

## P2（可排到后续小版本）

###7. 支持更多请求体识别场景

- 如：`params` 非首参、解构参数、更复杂函数声明形式。
- 用户价值：覆盖更多项目风格。
- 实现成本：中-高。

###8. 类型命名策略可配置

- 如：`{Base}Req/{Base}Resp`、后缀自定义。
- 用户价值：融入团队命名规范。
- 实现成本：中。

###9. Response生成策略可配置

- 如优先提取 `data` 或保留整体响应包裹结构。
- 用户价值：适配不同后端返回规范。
- 实现成本：中。

##4) 总结

当前 `0.1.3`版本整体可用、可发布、基础质量良好。当前最值得优先处理的是**发布包瘦身**与**CLI 在自动化场景下的稳定输出契约**，这两项投入小、收益高，建议优先落地。
